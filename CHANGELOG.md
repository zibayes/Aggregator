## Спринт 2 (04.04.2026 – 18.04.2026)

### 🔍 Анализ и выбор технологий

- Проведён сравнительный анализ библиотек оптического распознавания символов (OCR): **EasyOCR** и **Pytesseract**.
- По результатам тестирования принято решение об использовании гибридного подхода:
    - **Pytesseract** — для обработки координат и распознавания обозначений в формате DMS (градусы/минуты/секунды) за
      счёт лучшей работы с типографическими текстами.
    - **EasyOCR** — для общего сканирования текстовых блоков, демонстрирует более высокую точность обнаружения текста на
      сложных фонах.

### 🧪 Тестирование

- Разработан и интегрирован дополнительный набор модульных тестов для ключевых компонентов системы:
    - `test_acts_processing.py` — проверка обработки актов ГИКЭ.
    - `test_batch_file_organizer.py` — тестирование логики организации файлов при пакетной обработке.
    - `test_coordinates_extraction.py` — валидация алгоритмов извлечения координат из текстов и таблиц.
    - `test_open_lists_ocr.py` — оценка корректности OCR для открытых листов.
    - `test_scientific_reports_processing.py` и `test_tech_reports_processing.py` — тестирование обработки научных и
      научно-технических отчётов.
- Общее покрытие тестами ключевых модулей обработки увеличено, выявлены и исправлены незначительные ошибки в парсинге
  геоданных.

### 🧠 Обучение моделей

- Выполнено дообучение (fine-tuning) модели **DeepPavlov/rubert-base-case** на корпусе из 243 размеченных
  археологических текстов.
- На валидационной выборке достигнуты следующие показатели:
    - **accuracy** — 0.99
    - **F1-мера** — 0.75–0.76
    - **precision** — 0.88, **recall** — 0.71
- Модель демонстрирует сбалансированную динамику обучения без переобучения, что подтверждается стабилизацией validation
  loss.
- Запланировано обучение и сравнение дополнительных архитектур:
    - `sberbank-ai/ruBert-large`
    - `cointegrated/rubert-tiny2`
    - Цель — выбор оптимальной модели для дальнейшей интеграции в модуль извлечения именованных сущностей (NER).

### ⚙️ Интеграция

- Ранее разработанная модель сегментации учётных карт (Faster R-CNN) встроена в общий пайплайн обработки документов
  приложения.
- Обеспечена возможность последовательного выполнения: сегментация → OCR → извлечение атрибутов → сохранение
  структурированных данных.

### 📦 Результаты спринта

- Сформирована основа для модуля извлечения сущностей (NER) на основе fine-tuned модели rubert-base-case.
- Протестированы и подготовлены к использованию две OCR-библиотеки с разделением зон ответственности.
- Существующий пайплайн обработки документов дополнен блоком сегментации учётных карт.

## Спринт 3 (18.04.2026 – 02.05.2026)

### 🔧 Доработка конвейера обработки учётных карт

- Реализован двухэтапный алгоритм уточнения границ детектированных областей (refine_box_by_content):
    - После получения предсказаний Faster R-CNN каждая ограничивающая рамка расширяется на отступ (margin=40 пикселей).
    - Внутри расширенной области выполняется бинаризация (метод OTSU) и поиск контуров (cv2.findContours).
    - Рамка подрезается точно по границам фактического содержимого, что компенсирует неточности нейросетевой детекции и
      исключает лишние поля перед подачей в OCR.
    - Интегрирован модуль оптического распознавания символов (Pytesseract) в конвейер обработки учётных карт:
    - Для распознавания текста из уточнённых областей используется Pytesseract (выбран по результатам сравнительного
      анализа как более точный для типографских документов).
    - Реализована постобработка распознанного текста для извлечения атрибутивных данных (дат, номеров, координат).

### 🗺️ Конвертация геоданных в KML/KMZ

- Разработан модуль CoordinateProcessor для автоматизированной обработки координатной информации:
    - Поддержка парсинга координат в форматах DMS (градусы/минуты/секунды) с различными разделителями, десятичных
      градусов, а также с учётом типичных ошибок OCR.
    - Преобразование координат между системами (МСК, ГСК, WGS84) с использованием библиотеки pyroj и предопределённых
      проекций.
    - Автоматическое построение полигонов по извлечённым координатным парам, вычисление площади по формуле сферического
      избытка (для WGS84) или методу трапеций (для плоских проекций).
    - Экспорт в форматы KML/KMZ с группировкой объектов, цветовым кодированием и добавлением вычисленной площади в
      описание.

### 🧪 Тестирование

- Завершена разработка комплекта автоматизированных тестов:
    - Общее количество тестов — **612**, все успешно выполняются.
    - Покрытие кода для модулей представлений и обработки достигло **94%** (1860 из 1980 строк).
    - Непокрытые строки относятся к обработке редких краевых случаев и внешних зависимостей, которые сложно
      воспроизвести в тестовой среде (сетевые запросы, реальные файловые потоки). Эти участки проанализированы вручную и
      признаны низкорисковыми.
- Реализованные тесты охватывают:
    - модули извлечения координат (DMS-парсинг, конвертация);
    - обработку учётных карт (детекция, уточнение границ, OCR);
    - пакетную обработку файлов и пагинацию;
    - конвертацию геоданных в KML/KMZ.

### 📦 Результаты спринта

- Полностью завершён конвейер обработки отсканированных учётных карт: сегментация → уточнение границ → OCR → извлечение

- Реализован сквозной экспорт координатных данных в геопространственные форматы с автоматическим вычислением площади

- Обеспечено высокое покрытие кода тестами (94%), что гарантирует стабильность системы при последующих изменениях.
